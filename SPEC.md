# Wheel of Misfortune — Build Brief (SPEC)

A fair, fun, family chore-assignment web app. A person walks up to a phone or iPad, taps
their name (Netflix-profile style), flicks a prize wheel, and is handed a task. The wheel
**feels** random but is engineered so nobody gets stuck with the same job week after week
and the grim jobs get spread evenly. Completed work stacks into a per-person pile so
fairness is visible at a glance — equal-height piles mean an equal share of effort.

This document is the spec. Build it phase-by-phase (**§12**). The engine maths (**§4**) is
the heart — get it right and unit-tested before touching the server or UI.

> **Model note (read this first):** there are **two distinct quantities** in this app and
> they must never be merged. **Effort** is the currency we accumulate and balance — it is
> what the piles measure and what "fair share" is counted in. **Ick** (unpleasantness) is
> a *distribution constraint only*: we spread it evenly but we never reward it. Ick never
> adds to a pile, never becomes credit, and exists only to stop the wheel handing grim
> jobs to whoever has already done their share. There is no combined "burden" score.

## As-built delta (spec trued to reality — 2026-06-01)

This brief was followed closely; the sections below have been updated to match what shipped.
The key differences between the original spec and the built app:

- **New endpoint `POST /api/claim`** (§6, §7): take a *specific* on-wheel chore without
  spinning — to log one done after the fact, or to deliberately pick one. It is **not** a
  draw (φ is untouched, no reroll) but joins the normal done → pile flow, so a claimed
  weekly chore credits effort like any other. The UI exposes it as a second pill,
  **"+ Weekly chore"**, beside **"+ Daily chore"** (60/40), reusing the daily bottom-sheet.
- **`credit_remaining = quota − spent`** only (§4.2, §8). `banked` surplus is carried in
  state and shown as its own badge, but is **never folded into credit** (doing so once
  double-counted it). The dashed **fair-share line = the per-person quota** (the target),
  not a running average of pile heights.
- **`weekly_quota` is over in-play *on-wheel* tasks only** (§4.2) — off-wheel/daily effort
  never counts toward quota or piles.
- **`banked` is a real top-level state field** (§5), `{player: float}`, carried across
  rollovers. Claimed assignments also carry a `"claimed": true` flag.
- **`/api/done` accepts a `pending` task too** (§6), so a claim (and the quick-done path)
  can skip the separate accept tap. Only an already-`done` task is refused.
- **Rollover walks *each* elapsed week** (§6, §13): a multi-week idle gap drains banked
  surplus and breaks the at-quota streak week-by-week rather than collapsing into one. It
  prunes the closed `daily_log`, preserves `history`/`ledger`/`phi_cursor`, and leaves past
  weeks' `assignments` in place (it does not wipe them).
- **Sound is synthesised**, not sampled (§7): Web Audio oscillators + envelopes, so there
  are **no audio asset files** and it works fully offline. Server still names the `sfx` cue.
- **Accessibility + PWA hardening** (§7, §10): every kind renders an **icon glyph** (not
  colour alone), plus ARIA roles/live-regions, ≥44px touch targets, safe-area insets,
  `prefers-reduced-motion`, and a boot-error **Retry**. A **service worker** (`web/sw.js`)
  caches the app shell but **never caches `/api`**; icons are generated by a stdlib-only
  `deploy/make_icons.py` (no image library).
- **Added files** (§11): `pyproject.toml`, `Makefile`, `LICENSE`, `tests/test_api.py`,
  `web/sw.js`, `web/icons/icon-{180,192,512}.png`, `deploy/make_icons.py`, `backups/`.
- **Licensed `AGPL-3.0-or-later`** (`LICENSE`) — the network-use copyleft fits a
  self-hosted, shareable app.
- **Known gap** (§9): monthly/quarterly tasks seed `in_play:false` and there is **no
  "activate when due" mechanism yet**, so they never reach the wheel. `reseed()` exists as a
  function but is not yet wired to a CLI flag or make target.

## 2. What we're building (one paragraph)

A LAN-only web app served by a small **FastAPI** server running on a **Mac at home**. The
server is the single source of truth: it holds all state in an atomic JSON file, owns the
random-but-fair task-selection engine, and is the only thing that decides spin outcomes.
Phones and iPads are thin clients — they animate a prize wheel to whatever the server
already decided. Devices reach the server over Bonjour at `http://misfortune.local:8000`,
add it to the home screen as a PWA, and it stays up via `launchd` (or systemd on Linux). It
serves while the host is awake and sleeps with it; for always-on, run it on a low-power box (§10).

---

## 3. Architecture (locked)

- **Server:** FastAPI + Uvicorn, Python 3.11+. **Single worker** — state is an in-process
  guarded JSON file; multiple workers would each hold a divergent copy (load-bearing
  constraint, see §6).
- **State:** one JSON file, written atomically (temp file → `os.replace`), guarded by an
  `asyncio.Lock` around every read-modify-write.
- **Human-editable seed:** `family.yaml` (copied from `family.example.yaml`) is read **once**
  to populate `state.json` on first run. After that, `state.json` is the runtime ledger and
  is never hand-edited. See §5.1.
- **Outcome authority:** the spin result is decided **server-side**, before the wheel stops
  moving. Only the server sees everyone's history, advances the φ-stream once, and prevents
  two devices racing.
- **Networking:** Bonjour / mDNS. macOS auto-advertises `<hostname>.local`; all-Apple
  clients resolve it natively. Plain HTTP. **LAN-only — do not port-forward.**
- **Clients:** mobile-first static HTML/CSS/JS, installable as a PWA.
- **Keep-alive:** `launchd` LaunchAgent (RunAtLoad + KeepAlive) on macOS / a systemd `--user`
  unit on Linux. The host is **not** forced awake (no `caffeinate`) — it serves while awake and
  sleeps with the host; always-on is a low-power box's job (§10).

## 4. The fairness + randomness engine (the heart)

Three independent layers, composed per spin. Implement in `app/engine.py` as **pure
functions** with no I/O, fully unit-testable.

### 4.1 Effort and ick

Every task carries two independent numbers:

- **Effort** (the currency — accumulated, balanced, drawn as pile height):
  `XS=2, S=3, M=4, L=5`. Equal steps; hardest = 2.5× easiest. In the seed file people write
  the friendly token (`effort: M`); the loader maps it to points.
- **Ick** (the constraint — spread, never scored): `0=fine, 1=a bit grim, 2=grim, 3=truly grim`.

**There is no `burden`.** Effort is the ledger; ick only ever appears inside
`ick_balance_factor` (§4.3). Full dataset in **§9**.

### 4.2 Fair share & credit (counted in effort)

For a given week, with `P` players and a pool of in-play tasks:

```
weekly_quota       = sum(effort of in-play ON-WHEEL tasks) / P  # per person, effort points
spent_i            = sum(effort of tasks person i completed this week)
credit_remaining_i = weekly_quota - spent_i              # +ve = still owes, 0 = done, -ve = ahead
banked_i           = max(0, spent_i - weekly_quota)      # surplus, carried in state["banked"]
```

Quota is over **in-play on-wheel** tasks only — off-wheel daily chores carry no effort and
never count toward quota or the piles (§7, §9). **`banked` is carried across weeks (state
field) and shown as its own dashboard badge, but it is *not* folded back into
`credit_remaining`** — credit is always `quota − spent`, so a surplus is never double-counted
(once as a head-start credit, once as pile height). At a week boundary a player's surplus
drains against the new week's quota week-by-week (§13).

Ick is tracked per person **only within the period** (`ick_spent_i`) so §4.3 knows who's
had their share. It is not a quota anyone is asked to "meet" and never shows as credit.

**Quota is informational, never a gate.** Players self-pace — do three deep tasks or none in
an evening. `credit_remaining_i` drives the dashboard and the dashed fair-share line, but the
server never refuses a spin because someone has "met quota." Falling behind is surfaced
socially by the piles, not enforced by the API (§6: `/api/spin` is non-blocking).

### 4.3 Per-spin task weighting

When player `P` spins, weight every **unclaimed in-play task** `t`:

```
weight(t, P) = base(t) * cooldown_factor(P, t) * ick_balance_factor(P, t)
```

- **base(t) = 1** (uniform over the pool). Optional future knob: bias toward clearing
  high-effort tasks first.

- **cooldown_factor(P, t)** — the "I did that last week" killer. If `P` last did `t`
  `w` weeks ago:
  ```
  cooldown_factor = clamp(w / COOLDOWN_WEEKS, FLOOR, 1.0)
  ```
  `COOLDOWN_WEEKS = 3`, `FLOOR = 0.05` (never strictly impossible — avoids deadlock when the
  pool is tiny — just very unlikely). Recovers linearly to full eligibility.

- **ick_balance_factor(P, t)** — the *only* role ick plays. Spreads grim jobs. For a task
  with `ick > 0`, downweight it for a player already carrying more than the even share of
  ick this week:
  ```
  even_ick   = sum(ick_spent over players) / P          # the even share so far
  ick_excess = max(0, ick_spent_P - even_ick)
  ick_balance_factor = 1 / (1 + ick_excess * ICK_SENSITIVITY)   # for ick>0 tasks
                     = 1                                          # for ick==0 tasks
  ```
  `ICK_SENSITIVITY = 0.5`. Effort balance is the primary objective; ick-spread is a soft
  secondary nudge layered on top — they coexist, they are not jointly hard-constrained.

### 4.4 The φ low-discrepancy draw ("feels random")

Makes selection feel random while being unguessable and non-clustering. **It does not
replace the cooldown** — cooldown does "not the same task"; φ does "no learnable pattern,
no clustering."

Background (so nobody "simplifies" this back to uniform random): humans judge true
randomness as *rigged* because they expect short runs to mirror long-run averages (Tversky
& Kahneman, law of small numbers) and they over-alternate when producing "random" sequences
(Falk & Konold; Bar-Hillel & Wagenaar — switch probability ~0.6 vs the true 0.5). A
golden-ratio additive recurrence (a Weyl sequence, α = 1/φ) is a low-discrepancy /
quasirandom stream: maximally spread, aperiodic, unguessable (Roberts 2018). Feels random,
doesn't clump.

```python
PHI_ALPHA = 0.6180339887498949        # = 1/φ
# phi_cursor is persisted in state, seeded with a random float in [0,1) at first run.
def draw(weights, phi_cursor):
    phi_cursor = (phi_cursor + PHI_ALPHA) % 1.0     # advance once per spin
    total = sum(weights.values())
    target = phi_cursor * total
    acc = 0
    for task_id, w in sorted(weights.items()):      # stable order → deterministic CDF
        acc += w
        if target < acc:
            return task_id, phi_cursor
    return sorted(weights)[-1], phi_cursor           # float-safety fallback
```

The cursor is persisted and advanced **exactly once per `draw()`** so the stream is shared
across devices and survives restarts. **Lifecycle (resolved):** the cursor runs
**continuously across weeks** — rollover never resets it. A **reroll is a fresh `draw()`**
over the pool with the rejected task excluded from that draw (it returns to the pool for
everyone else); the reroll therefore advances the cursor once more. So an accepted first spin
advances it once; a spin + one reroll advances it twice.

### 4.5 Tunable knobs (in `config`; no logic change to tweak)

`COOLDOWN_WEEKS=3`, `FLOOR=0.05`, `ICK_SENSITIVITY=0.5`, `PHI_ALPHA=1/φ`,
`REROLLS_PER_SPIN=1`, `players`, `week_start`, and the **effort scale map**
(`XS=2,S=3,M=4,L=5`).

## 5. Data model (`state.json`)

```jsonc
{
  "version": 1,
  "config": {
    "players": ["Martin", "Sarah", "Alice"],
    "player_tints": { "Martin": "#5B6B7A", "Sarah": "#7A8B5E", "Alice": "#8A6F8E" },
    "effort_scale": { "XS": 2, "S": 3, "M": 4, "L": 5 },
    "cooldown_weeks": 3,
    "cooldown_floor": 0.05,
    "ick_sensitivity": 0.5,
    "phi_alpha": 0.6180339887498949,
    "rerolls_per_spin": 1,
    "week_start": "monday",
    "kinds": {
      "surfaces": { "label": "Surfaces & dishes", "hue": "#E0A100", "icon": "sparkle" },
      "floors":   { "label": "Floors",            "hue": "#1F8A8A", "icon": "broom"  },
      "laundry":  { "label": "Laundry",           "hue": "#3D6FD6", "icon": "shirt"  },
      "waste":    { "label": "Waste",             "hue": "#C2410C", "icon": "trash"  },
      "wet":      { "label": "Wet & sanitise",    "hue": "#A01A58", "icon": "spray"  }
    }
  },
  "phi_cursor": 0.0,                         // seeded random at first run
  "current_week": "2026-W23",
  "tasks": [                                 // seeded from family.yaml → §9
    { "id": "clean_toilet", "room": "Bathroom", "kind": "wet",
      "name": "Disinfect, de-limescale & clean toilet",
      "freq": "weekly", "effort": "M", "ick": 3, "in_play": true, "on_wheel": true }
    // on_wheel derived from freq: weekly/fortnightly/monthly/quarterly = true (deep → wheel);
    // daily and 2–3×/week = false (frequent → daily bonus button, §7). Override with `wheel:`.
  ],
  "assignments": {
    "2026-W23": {
      "clean_toilet": { "player": "Alice", "status": "done",
                        "spun_at": "...", "accepted_at": "...", "done_at": "...",
                        "rerolls_used": 0 }
    }
  },
  "history": [                               // append-only completion log (wheel tasks only)
    { "week": "2026-W23", "task_id": "clean_toilet", "player": "Alice",
      "effort": 4, "ick": 3, "done_at": "..." }
  ],
  "daily_log": {                             // off-wheel daily chores — counts only, never effort
    "2026-W23": { "Martin": [ { "task_id": "wipe_counters", "at": "..." } ] }
  },
  "recent_spins": {},                        // {spin_token: task_id} idempotency cache, last ~32
  "banked": {                                // surplus effort carried across weeks (a badge,
    "Martin": 0.0, "Sarah": 0.0, "Alice": 0.0 //  not folded into credit_remaining — §4.2)
  },
  "ledger": {                                // lifetime cumulative, for stats
    "Martin": { "lifetime_effort": 0, "lifetime_ick": 0, "weeks_at_quota": 0, "daily_taps": 0 }
  }
}
```

`status` ∈ `pending` → `accepted` → `done` (a claimed or quick-done task may go straight
`pending → done`; only an already-`done` task is refused). A claim also stamps the assignment
`"claimed": true`. Cooldown last-done lookups read from `history`.
Note `effort` is stored as the **token** on tasks (resolved via `effort_scale`) and as the
**resolved integer** on history rows.

### 5.1 The seed file (`family.yaml`)

`family.example.yaml` is the canonical human-editable definition (chores, players, kinds,
settings) in commented YAML — one aligned line per chore, friendly tokens, defaults
implicit. On first run the loader copies/reads `family.yaml` into `state.json`. Re-seeding
rebuilds task definitions **without** wiping `history`, `ledger`, `banked` or `phi_cursor`
(match on task `id`; auto-slug the name when `id` is omitted) — implemented as `seed.reseed()`,
though a `--reseed` flag / make target to invoke it isn't wired yet (see the as-built notes).

**Validation must be friendly** (acceptance criterion): on a malformed seed, report the
line and a human fix ("line 47: unknown kind 'bins' — did you mean one of
surfaces/floors/laundry/waste/wet?"), never a raw stack trace. This is the single biggest
factor in non-technical adoption.

## 6. API

All JSON. All mutating calls go through the `asyncio.Lock` + atomic save.

| Method | Path | Body | Returns | Notes |
|--------|------|------|---------|-------|
| GET | `/api/state` | — | full state | client hydrate |
| GET | `/api/dashboard` | — | credits (effort), ick spread, streaks, gaps | §8 |
| POST | `/api/spin` | `{player, spin_token?}` | `{task, weight_breakdown, credit_remaining}` | picks from the unclaimed in-play **on-wheel** pool, advances φ once, marks `pending`. **Non-blocking** — always allowed while the pool is non-empty; returns 409 `{error:"pool_empty"}` only when nothing is left. Idempotent per `spin_token`. |
| POST | `/api/claim` | `{player, task_id}` | `{task, credit_remaining, claimed:true}` | take a **specific** on-wheel chore *without* spinning (log one done after the fact, or pick deliberately). **Does not advance φ** and has no reroll; marks `pending` + `claimed`. 400 if the task is off-wheel/not in play, 409 if already taken this week. |
| POST | `/api/reroll` | `{player, task_id}` | `{task, weight_breakdown}` | 403 if no rerolls left (`rerolls_used ≥ rerolls_per_spin`), 404 if no such pending task; **fresh φ draw excluding `task_id`**; old task returns to pool; 409 if the pool is then empty (original handed back, reroll consumed) |
| POST | `/api/accept` | `{player, task_id}` | `{ok}` | `pending → accepted` |
| POST | `/api/done` | `{player, task_id}` | `{ok, credit_remaining, fireworks:true}` | `accepted → done` **or** `pending → done` (claims & quick-done skip accept); append history, recompute. Refuses an already-`done` task. |
| POST | `/api/daily` | `{player, task_id}` | `{ok, daily_count, sfx}` | logs a **daily/frequent** (off-wheel) chore. Increments `daily_log[week][player]`; **never** touches effort, quota, piles, `history`, or φ. Returns an `sfx` cue key for the client to play. Idempotent per `(player, task_id, UTC date)`. |
| POST | `/api/week/rollover` | — | `{week}` | close stale week(s), carry banked surplus, prune the closed `daily_log`; **leaves past `assignments` in place**. Also auto-runs lazily when `current_week` != today's ISO week; a multi-week gap is accounted week-by-week (§13). |

GET `/api/state` and `/api/dashboard` are read-only: they roll the week over (a write) **only**
when it is actually stale, so a routine poll is a single read, not an fsync+rewrite.

**Concurrency:** single Uvicorn worker; every write wrapped in the lock; atomic file
replace; short-lived idempotency token on `/api/spin` so a double-tap doesn't double-assign.

## 7. Frontend / UX

Mobile-first. Two screens: **Spin** and **Dashboard**. (See `prototype/` for the feel.)

### Spin loop
1. **Pick name** — Netflix-style profile picker: three big avatar/initial chips in each
   player's *muted tint*. Identity is initial + muted tint, never a saturated hue.
2. **Flick the wheel** — swipe-to-flick. Flick velocity controls *rotations + duration
   only*, never the outcome (server already decided). Hard ≈5 turns, soft ≈2, both settle
   on the server's sector with momentum + small overshoot wobble. Animate-to-target.
3. Wheel **sectors are the candidate pool** (cap ~8), coloured by kind. (Prototype uses
   equal-angle sectors for label legibility; weighted-size sectors are an option.)
4. **Task card** → `I accept` / `Reroll (1 left)`.
5. Accept → `Mark done when finished`.
6. **Done → fireworks** + the completed tile drops onto the player's pile (§8).

### Daily & weekly chore buttons (off the wheel / off the random)
Below the wheel, a row of **two pills (60/40)**, both calm so neither competes with the wheel:
- **"+ Daily chore"** (paper/ink) → a bottom-sheet of the daily/frequent off-wheel chores →
  one tap logs it with a **sound effect** + a confetti pop. No spin, no accept/reroll. Daily
  taps are a separate counter: a happy noise and a weekly tally, never effort, piles, or
  fairness.
- **"+ Weekly chore"** (gold-outline) → reuses the same bottom-sheet, listing the available
  in-play **on-wheel** chores (with effort), to **claim** one without spinning (`/api/claim`)
  — for a chore done after the fact, or a deliberate pick when you're not in the mood for
  random. A claimed task shows the normal card with **reroll hidden** and goes straight to
  **"I'm done"**; it credits effort onto the pile exactly like a spun task.

**Sound:** unlock a single `AudioContext` on the first user gesture (iOS Safari blocks audio
until then). Cues are **synthesised on the fly** (oscillator + gain envelope), *not* loaded
from sample files — so there are **no audio assets** and it works fully offline; pitch is
jittered slightly so repeats don't grate. A persistent **mute** toggle (stored in
`localStorage`, default on) governs all sound — the wheel tick, the chore noises, the done
chime. The server names the cue (`sfx` in the `/api/daily` reply); the client plays it.
`prefers-reduced-motion` is honoured: confetti and the long wheel spin are dropped (the wheel
still resolves to the result), sound still plays.

### 7.1 Colour & visual encoding
Two categorical colour systems kept apart by **saturation**:
- **Task kind → saturated hue** (primary; on wheel sectors, cards, pile tiles). Every task
  has one kind, grouped by shared tools/products/sequence, not room.
- **Player → muted tint + initial** (name picker, pile bases, dashboard, confetti).

Colour is never the only signal — pair kinds with an icon + label, players with an initial.
**Effort and ick are not colour-coded:** effort shows as an XS/S/M/L badge, ick as a row of
0–3 grim pips.

| Kind | Hue | In it | Grouped because |
|------|-----|-------|-----------------|
| Surfaces & dishes | amber `#E0A100` | dust/wipe surfaces, bannisters, stairs, counters, cupboards, shelving, dishwasher | dry cloth, no special products |
| Floors | teal `#1F8A8A` | sweep/hoover/mop of dry rooms (lounge, bedrooms, kitchen) | broom + mop + bucket; done last |
| Laundry | blue `#3D6FD6` | wash/hang/fold, towels, bedlinen, to-machine | the machine pipeline |
| Waste | rust `#C2410C` | empty kitchen + bathroom bins | hands-on with rubbish |
| Wet & sanitise | berry `#A01A58` | bathroom sink/toilet/cistern/floor, fridge interior | disinfectant/limescale; **cloths quarantined from kitchen** |

**Deliberate call:** the **bathroom floor is "Wet & sanitise," not "Floors"** — colour
follows the bucket/quarantine, not the motion. Flip to "Floors" to make colour follow motion.

### Design direction
Warm dark **"fairground at night"**: deep aubergine/midnight base, gold accents. Five
saturated kind hues carry the energy; three muted player tints sit underneath. Characterful
display font (e.g. Fraunces) + a monospace for all numbers (JetBrains Mono — ledger feel).
The wheel should feel like a physical prize wheel: real momentum, satisfying deceleration,
tactile flick. Confetti only on `done`. Avoid Inter/Roboto/system fonts and purple-on-white.

## 8. Dashboard + the piles

**The piles (the headline visual).** Three vertical columns, side by side, one per player.
Each completed task is a tile stacked bottom-up; **tile thickness = effort points** (so a
mopping job is visibly thicker than a wiped ledge). **Equal-height piles = an equal share of
effort done.** Tiles coloured by kind (each tile also carries its kind **icon**, so the
piles aren't colour-only). A dashed "fair share" line sits at the **per-person weekly quota**
— the target each pile aims for — so anyone whose pile is below it visibly still owes effort
this week (the gap from a pile's top to the line *is* that player's `credit_remaining`). This
is the fairness signal a child can read without numbers.

**Per person (this week):** credit remaining (`weekly_quota - spent`, in effort), banked,
jobs count, ick-spread (how this week's grim work is distributed — informational, not a
target), and **daily taps** (off-wheel chores logged this week — a separate fun tally, never
part of the pile or quota).

**Per person (lifetime):** lifetime effort, lifetime ick, **good streak** (consecutive
weeks at quota).

**Per task:** **gap** — "last done by {player}, {n} weeks ago" — the visible receipt that
makes the cooldown legible and answers "but I did that last week."

Compute all of this in `/api/dashboard` from `history` + `assignments`; the client renders.

## 10. Deployment runbook (`deploy/runbook.md`)

**Install**
```bash
cd wheel-of-misfortune
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt        # fastapi, uvicorn[standard], pyyaml
cp family.example.yaml family.yaml     # then edit family.yaml
```

**Run (single worker)**
```bash
.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1
```
The host is not forced awake; it serves while awake and sleeps with the host. For a wake
window use `pmset repeat`; for always-on, a low-power box (`make run` / `make deploy`).

**Bonjour** — macOS auto-advertises `<hostname>.local` (set in System Settings → General →
Sharing → *Local hostname*). For a friendly name, set it to `misfortune` →
`http://misfortune.local:8000`. All Apple devices resolve `.local` natively.

**launchd (macOS)** — `deploy/com.storey.misfortune.plist` → `~/Library/LaunchAgents/`.
Absolute paths; `WorkingDirectory`, `RunAtLoad`, `KeepAlive`, `StandardOut/ErrorPath` to a
log. No `caffeinate` — the Mac isn't forced awake. Load with `launchctl load -w …`.
`deploy/install.sh` copies the plist, substitutes paths, loads it.

**systemd (Linux)** — the app code is OS-neutral; only this keep-alive layer differs.
`deploy/wheel-of-misfortune.service` is a systemd `--user` unit (`Restart=always` ≈ KeepAlive;
`loginctl enable-linger` ≈ RunAtLoad). The **same** `deploy/install.sh` detects the OS and
installs launchd or systemd accordingly. The app works over the plain LAN IP with **nothing
extra installed**; `.local` resolution is built-in on macOS/iOS, and on Linux it's an optional
convenience (install Avahi only if you want the *host* to answer to `<hostname>.local` too).
See `deploy/runbook.md` §9 for the full Mac→Linux map.

**Anti-sleep (belt and braces)**
```bash
sudo pmset -a sleep 0 disksleep 0          # macOS  (Linux servers don't suspend; laptop → mask sleep.target)
```

**PWA** — `web/manifest.json` (standalone, theme `#171022`), `apple-touch-icon`, and the
`apple-mobile-web-app-capable` meta. A **service worker** (`web/sw.js`) precaches the app
shell for an instant launch on a flaky LAN but **never caches `/api`** (the server stays the
sole source of truth); bump its `VERSION` to ship front-end changes. Icons
(`web/icons/icon-{180,192,512}.png`) are generated by `deploy/make_icons.py` — stdlib only,
no image library (`make icons`). Add to Home Screen in Safari → full-screen app.

**Security:** LAN-only. Bind `0.0.0.0`, **do not** port-forward. (Future remote access:
Tailscale — out of scope.)

**License:** the project is released under **`AGPL-3.0-or-later`** (`LICENSE`) — its
network-use copyleft suits a self-hosted app meant to be tinkered with and shared.

## 13. Tests / acceptance criteria

The engine is pure and seedable — assert behaviour, not vibes:

- **Determinism:** same `phi_cursor` + state → identical spin sequence.
- **Cooldown:** 500 spins / 3 players / many weeks → no task re-assigned to the same player
  inside `COOLDOWN_WEEKS` (except rare `FLOOR`-forced cases when the pool is tiny — assert
  rare).
- **Effort fairness:** per-player cumulative **effort** converges within a tolerance band of
  fair share.
- **Ick spread:** per-player cumulative **ick** stays within tolerance — no player collects a
  disproportionate share of grim jobs. (Ick never appears in the effort assertions.)
- **"Feels random":** lag-1 repeat rate of the φ-driven selection is **lower** than a
  uniform-random baseline over the same pool. Optionally a runs test.
- **Seed validation:** a malformed `family.yaml` yields a line-numbered, human-readable
  error, not a stack trace.
- **Concurrency:** two simultaneous `/api/spin` never double-assign and advance φ once each.
- **Rollover:** week boundary carries `banked` surplus and preserves `history` + `ledger` +
  `phi_cursor`; a **multi-week** gap is accounted week-by-week (idle weeks drain the bank and
  break the at-quota streak); the closed `daily_log` is pruned. (As-built does **not** wipe
  past weeks' `assignments`.)
- **Claim:** `/api/claim` assigns a specific on-wheel task **without** advancing φ; it then
  credits effort onto the pile via the normal done flow; off-wheel → 400, already-taken → 409.
- **Equal participation → equal piles:** with everyone participating equally, per-player
  cumulative effort (pile height) converges to within a few percent over many weeks.
